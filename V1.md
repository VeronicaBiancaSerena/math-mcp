# math-mcp V1 优化方案

本文档针对实际使用中暴露的可发现性、错误自纠正、有限枚举、约束验证和输出冗长度问题，给出 V1 优化方案。V1 的目标不是扩大数学能力边界，而是让 agent 更容易选对 operation、传对 schema，并在失败后能低成本自我修正。

## 1. 核查结论

### 1.1 外部 MCP / schema 设计核查

- MCP 工具是 model-controlled 的可发现接口，工具定义依赖 `name`、`description`、`inputSchema` 等元数据帮助模型选择和调用。
- MCP 工具执行错误应返回 agent 可用的、可自纠正的反馈；输入校验错误、业务逻辑错误应尽量通过 tool execution error 表达，而不是只给模糊失败。
- JSON Schema 的 `enum` 适合描述固定可选值，例如 operation 名、有限 mode、domain kind。
- 符号系统对“带约束恒等验证”的支持通常是分层的：显式代入/参数化可给强证明；自由变量采样只能给反例或证据；任意约束上的全局证明不能默认承诺。

### 1.2 本地复现结论

- `algebra_compute(operation="simplify")` 当前返回 `UNSUPPORTED_OPERATION`，错误为 `unknown operation 'simplify' for tool 'algebra_compute'`，没有推荐 `simplify_expression`。
- `trigonometry_compute(operation="simplify")` 当前同样返回 `UNSUPPORTED_OPERATION`，没有推荐 `trig_simplify`。
- `finite_enumeration` 需要顶层 `domains`，如果把 domain 放进 `payload.domains`，当前会忽略该字段并报缺少变量 domain。
- `math_capabilities()` 默认输出完整 registry、payload schema、limits、examples，信息完整但很长，不适合作为 agent 高频检索入口。

## 2. V1 目标

1. 降低 operation 名称误用率：常见直觉名能被解析为规范名。
2. 提高失败可恢复性：未知 operation、错误 domain 位置、缺失 domain 时返回推荐修正。
3. 降低能力发现成本：提供只列 tool 和 operation 的摘要模式。
4. 明确有限枚举 schema：让有限集合域和有界整数域的写法在 capabilities 与文档中直接可见。
5. 明确带约束验证/优化的能力边界：不把自由变量采样误报为约束下反例或证明。
6. 保持向后兼容：规范 operation 名、默认 full capabilities 结构、`ToolResult` 主字段不破坏现有调用。

## 3. 问题到方案映射

| 问题 | 当前表现 | V1 方案 | 验收标准 |
| --- | --- | --- | --- |
| operation 名不直观 | `simplify` 失败，真实名是 `simplify_expression` / `trig_simplify` | 增加 per-tool alias map，调用时解析到规范 operation | `simplify` 在代数和三角入口都能成功执行，并在 metadata 中记录 alias 解析 |
| 错误提示弱 | 只返回 `unknown operation` | 未命中时返回 `suggested_operations`、可用 operation 摘要和 `Did you mean ...` 文案 | `simlify` / `simplify` 等失败路径能给 1-3 个候选 |
| capabilities 太长 | 默认输出完整 schema | `math_capabilities(mode="summary")` 只返回工具名、operation 名、alias | summary 输出不含 payload schema/default_limits，体积明显小于 full |
| 有限枚举难用 | domain 必须顶层传，但 schema 只展示 payload | 增加 full call example、domain_schema、缺失/错位 domain 的专项错误提示 | capabilities 能直接看到有限集合域和整数区间域示例 |
| 约束下恒等验证不可靠 | `check_identity` 按自由变量采样 | 新增显式约束接口，优先参数化/代入证明，无参数化时只给 evidence 或 unsupported | 椭圆约束例子不会产生自由变量无关反例 |
| 工具输出冗长/偶发慢 | agent 频繁拉全量 capabilities；错误调用仍消耗检索成本 | 默认建议 agent 先拉 summary；operation 查找和 alias 解析在进入后端前完成 | 未知 operation 不进入后端/子进程，快速返回结构化建议 |

## 4. Capabilities 摘要模式

### 4.1 API 形状

保持现有参数兼容，新增 `mode`：

```python
math_capabilities(
    include_experimental: bool = False,
    include_disabled: bool = False,
    mode: str = "full",  # "summary" | "full"
)
```

`mode="full"` 保持现有返回结构。`mode="summary"` 返回轻量结构：

V1 决策：默认值必须继续是 `mode="full"`，避免破坏现有客户端和测试；需要低成本检索时由 agent 显式请求 `mode="summary"`。

```json
{
  "server": "math-mcp",
  "schema_version": "1.0",
  "capabilities_version": "1.0",
  "mode": "summary",
  "public_tools": {
    "algebra_compute": {
      "kind": "compute",
      "operations": [
        "simplify_expression",
        "expand_expression",
        "factor_expression",
        "solve_equation"
      ],
      "aliases": {
        "simplify": "simplify_expression",
        "factor": "factor_expression",
        "solve": "solve_equation"
      }
    },
    "trigonometry_compute": {
      "kind": "compute",
      "operations": [
        "trig_simplify",
        "trig_expand",
        "trig_reduce",
        "trig_identity_check"
      ],
      "aliases": {
        "simplify": "trig_simplify",
        "expand": "trig_expand",
        "reduce": "trig_reduce"
      }
    }
  }
}
```

### 4.2 Agent 使用建议

- 不确定有哪些工具或 operation 时，先调用 `math_capabilities(mode="summary")`。
- 已选定 operation 但不确定 payload 时，再调用 `math_capabilities(mode="full")` 查看该 operation 的 schema 和 example。
- 后续可继续增加 `tool` / `operation` 过滤参数，但不放入 V1 必做范围。

## 5. Operation alias 策略

### 5.1 原则

- alias 只在单个 public tool 内解析，不做跨 tool 自动跳转。
- alias 必须无歧义；同一 tool 内一个 alias 只能指向一个规范 operation。
- 成功调用时，`metadata.operation` 记录规范 operation，另加 `metadata.requested_operation` 和 `metadata.operation_alias_resolved=true`。
- capabilities summary 暴露 alias；full mode 可在 operation spec 或 tool level 暴露 alias。
- alias 不替代规范名。文档、测试 golden case、registry 主键仍使用规范 operation 名。

### 5.2 V1 alias 清单

`algebra_compute`：

```json
{
  "simplify": "simplify_expression",
  "expand": "expand_expression",
  "factor": "factor_expression",
  "cancel": "cancel_expression",
  "together": "together_expression",
  "solve": "solve_equation",
  "roots": "polynomial_roots",
  "groebner": "groebner_basis"
}
```

`trigonometry_compute`：

```json
{
  "simplify": "trig_simplify",
  "expand": "trig_expand",
  "reduce": "trig_reduce",
  "rewrite": "trig_rewrite",
  "solve": "solve_trig_equation",
  "identity_check": "trig_identity_check",
  "check_identity": "trig_identity_check"
}
```

其他 tool 暂不批量加 alias，避免引入误导。V1 后根据真实失败日志再扩展。

## 6. 错误提示与推荐 operation

### 6.1 未知 operation 返回结构

未知 operation 仍返回：

- `ok=false`
- `status="unsupported"`
- `error_code="UNSUPPORTED_OPERATION"`

但增强 `error` 和 `metadata`：

```json
{
  "error": "unknown operation 'simplify' for tool 'algebra_compute'. Did you mean 'simplify_expression'?",
  "metadata": {
    "public_tool": "algebra_compute",
    "operation": "simplify",
    "operation_state": "unknown",
    "suggested_operations": ["simplify_expression"],
    "available_operations": [
      "simplify_expression",
      "expand_expression",
      "factor_expression",
      "solve_equation"
    ],
    "suggestion_source": "alias"
  }
}
```

### 6.2 推荐来源优先级

1. alias 精确匹配。
2. 同 tool 内 operation 的 close match，例如 `difflib.get_close_matches`。
3. 同 tool 内按关键词映射，例如 `simplify` 匹配包含 `simplify` 的 operation。
4. 没有候选时只返回 `available_operations` 的前若干项，避免塞满输出。

## 7. finite_enumeration schema 与示例

### 7.1 当前正确调用形状

`domains` 是 compute tool 的顶层参数，不属于 `payload`：

```json
{
  "operation": "finite_enumeration",
  "payload": {
    "predicate": "Eq(x + y, 3)",
    "variables": ["x", "y"],
    "collect_witnesses": true
  },
  "domains": [
    {"variable": "x", "kind": "integer", "lower": "0", "upper": "3"},
    {"variable": "y", "kind": "integer", "lower": "0", "upper": "3"}
  ]
}
```

有限集合域写法：

```json
{
  "operation": "finite_enumeration",
  "payload": {
    "predicate": "Eq(x**2, 1)",
    "variables": ["x"]
  },
  "domains": [
    {"variable": "x", "kind": "finite", "values": ["-1", "0", "1"]}
  ]
}
```

### 7.2 capabilities 增强

在 `finite_enumeration` 的 operation capability 中增加：

```json
{
  "requires_domains": true,
  "domain_schema": {
    "type": "array",
    "items": {
      "type": "object",
      "required": ["variable", "kind"],
      "properties": {
        "variable": {"type": "string"},
        "kind": {"type": "string", "enum": ["finite", "integer"]},
        "values": {"type": "array", "items": {"type": "string"}},
        "lower": {"type": "string"},
        "upper": {"type": "string"}
      }
    }
  },
  "example_request": {
    "operation": "finite_enumeration",
    "payload": {
      "predicate": "Eq(x + y, 3)",
      "variables": ["x", "y"]
    },
    "domains": [
      {"variable": "x", "kind": "integer", "lower": "0", "upper": "3"},
      {"variable": "y", "kind": "integer", "lower": "0", "upper": "3"}
    ]
  }
}
```

### 7.3 错误提示增强

当 `finite_enumeration` 缺少 domain：

```text
finite_enumeration requires a finite or bounded-integer domain for 'x'. Pass domains as a top-level argument, e.g. domains=[{"variable":"x","kind":"integer","lower":"0","upper":"3"}].
```

当检测到 `payload.domains` 但顶层 `domains` 为空：

```text
finite_enumeration received domains inside payload, but domains must be a top-level argument. Move payload.domains to the tool argument named domains.
```

V1 决策：先返回明确错误，不自动搬运 `payload.domains`。原因是当前架构明确把 `domains` 作为跨 operation 顶层字段，自动搬运可能掩盖客户端调用形状错误。若真实使用仍频繁失败，V1.1 再考虑兼容性搬运并加 warning。

## 8. 带约束恒等验证

### 8.1 新 operation

新增：

```text
verification_compute(operation="check_identity_constrained")
```

建议 payload：

```json
{
  "left": "x**2/4 + y**2/3",
  "right": "1",
  "variables": ["x", "y"],
  "constraints": [
    {"relation": "==", "left": "x**2/4 + y**2/3", "right": "1"}
  ],
  "parameterization": {
    "variables": ["t"],
    "substitutions": {
      "x": "2*cos(t)",
      "y": "sqrt(3)*sin(t)"
    }
  }
}
```

### 8.2 语义分层

1. 显式参数化路径：先验证参数化满足约束，再把 substitutions 代入 `left-right`，调用现有 `check_identity`。若化简为 0，可返回 `proved_by_symbolic_simplification`。
2. 显式代入路径：若 payload 给出 `substitutions`，直接在约束消元后验证，证明等级同现有 `check_identity`。
3. 约束采样路径：若没有参数化/代入，只能在约束满足的样本上搜索反例。找到反例可返回 `disproved_by_counterexample`；未找到只能返回 `numeric_evidence_only`。
4. 不支持的约束：返回 `unsupported`，建议新增 `error_code="CONSTRAINT_UNSUPPORTED"`。如果不想扩展 enum，V1 可临时复用 `DOMAIN_UNSUPPORTED`，但文案必须说明是不支持该约束证明方式。

### 8.3 关键约束

- 禁止在 `check_identity_constrained` 中退化为自由变量网格采样。
- 结果 metadata 必须记录 `constraint_mode`：
  - `parameterized_symbolic`
  - `substitution_symbolic`
  - `constrained_sampling`
  - `unsupported`
- 证明 certificate 必须包含：
  - 原始约束；
  - 使用的 substitution/parameterization；
  - 约束满足性验证摘要；
  - 代入后的恒等式。

## 9. 约束优化接口

### 9.1 新 operation

新增：

```text
calculus_compute(operation="constrained_optimize")
```

建议 payload：

```json
{
  "objective": "x*y",
  "variables": ["x", "y"],
  "goal": "max",
  "constraints": [
    {"relation": "==", "left": "x**2/4 + y**2/3", "right": "1"}
  ],
  "method": "symbolic_lagrange"
}
```

### 9.2 语义分层

- `method="symbolic_lagrange"`：返回候选点、目标值、约束检查；只有能证明候选覆盖全部可行极值时，`certainty` 才能高于 `evidence`。
- `method="numeric"`：可用 SciPy 约束优化，`certainty="evidence"`，必须返回起点、约束残差、收敛信息。
- `parameterization` 存在时，先降维再用现有 `numeric_optimize` 或符号求导。

V1 不承诺任意多约束全局优化证明；默认要诚实地区分 exact/evidence。

## 10. 输出冗长与响应速度

1. `math_capabilities(mode="summary")` 是主要降噪入口。
2. 未知 operation、alias 解析、domain 错位检查必须在进入后端和子进程之前完成。
3. full capabilities 保持完整，但可在后续增加 `tool` / `operation` 过滤：

```python
math_capabilities(mode="full", tool="discrete_compute", operation="finite_enumeration")
```

4. 错误建议最多返回 3 个 `suggested_operations`，`available_operations` 最多返回当前 tool 的前 20 个，防止错误结果本身变长。

## 11. V1 实施顺序

1. 文档和测试用例先行：新增/更新 tests 覆盖 alias、summary、错误建议、finite_enumeration domain 示例、约束验证不退化自由采样。
2. 在 registry 或独立模块增加 per-tool alias map，并让 capabilities summary 读取同一份 alias 数据。
3. 修改 dispatcher：在 `get_spec` 前解析 alias；未知 operation 时生成建议。
4. 修改 capabilities：支持 `mode="summary"`，full mode 兼容原结构。
5. 增强 finite_enumeration：增加 capabilities 元数据和缺失/错位 domain 错误。
6. 新增 `check_identity_constrained`，先支持参数化/显式代入路径；采样路径只给 evidence。
7. 新增 `constrained_optimize`，先支持参数化降维和数值 evidence；符号 Lagrange 可作为第二阶段。

## 12. V1 验收清单

- `algebra_compute(operation="simplify", payload={"expression":"x+x","variables":["x"]})` 成功，metadata 记录 resolved operation 为 `simplify_expression`。
- `trigonometry_compute(operation="simplify", payload={"expression":"sin(x)**2+cos(x)**2","variables":["x"]})` 成功，metadata 记录 resolved operation 为 `trig_simplify`。
- `algebra_compute(operation="simlify", ...)` 返回 `UNSUPPORTED_OPERATION`，metadata 包含 `suggested_operations=["simplify_expression"]`。
- `math_capabilities()` 默认仍返回 full capabilities；`math_capabilities(mode="summary")` 只返回 tool、operation、alias，不返回 `payload_schema`、`default_limits`。
- `discrete_compute(finite_enumeration)` 的 capabilities 能看到完整 `example_request`，其中 `domains` 位于顶层。
- 把 `domains` 放进 `payload` 时，返回明确迁移提示。
- 椭圆约束下的恒等验证必须使用参数化/代入，或明确返回 evidence/unsupported；不得按自由变量给无关反例。
- 未知 operation 与 alias 解析路径不进入后端或 sandbox 子进程。

## 13. 非目标

- 不把 math-mcp 做成自然语言数学求解器。
- 不支持 LaTeX 直接输入。
- 不承诺任意代数簇上的全局恒等证明。
- 不用数值采样冒充证明。
- 不破坏已有规范 operation 名和 full capabilities 结构。
